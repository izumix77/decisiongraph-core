# DecisionGraph Core — Migration Guide v0.3 → v0.4

**Normative Reference:** Constitution v0.4 RC
**Supersedes:** Migration Guide v0.2 → v0.3

---

## Why v0.4?

v0.3 stored `Superseded` as a Node attribute.
This was a category error.

A Node is a **proposition** — a record that a decision was made,
by a specific author, at a specific time.
A proposition cannot contain its own negation or replacement state.
Whether that proposition has been replaced is a **relational fact**
between Nodes, not an internal attribute of any single Node.

Storing `status: "Superseded"` on a Node:

- conflated the proposition with its fate
- required mutation of committed Nodes to express supersession
- made `replayAt` ambiguous: stored state diverged from structural truth
- forced validators to read Node fields instead of topology

v0.4 resolves this completely.

> **Core claim of v0.4:**
> Supersession is a relation, not a state.
> `effectiveStatus` is derived from graph topology — never stored.

## Breaking Changes Summary

| # | Area | v0.3 | v0.4 |
|---|---|---|---|
| 1 | Node schema | `status: Active\|Superseded\|Deprecated` | **`status` field removed** |
| 2 | Node supersession | stored `node.status = "Superseded"` | derived via `effectiveStatus(store, nodeId)` |
| 3 | Deprecated concept | Core-level WARN | **removed from Core** |
| 4 | EdgeType | includes `overrides` | **`overrides` removed** |
| 5 | Edge status | `Active\|Superseded\|Deprecated` | **`Deprecated` removed** — binary only |
| 6 | Policy interface | `validateOperation(graph, op)` | `validateOperation(store, graphId, op)` |
| 7 | Policy interface | `validateGraph(graph)` | `validateStore(store)` |
| 8 | ApplyResult | `.graph` | **`.store`** |
| 9 | replay signature | `replay(ops[], policy): Graph` | `replay(logs: GraphLog[], policy): GraphStore` |
| 10 | replayAt signature | `replayAt(ops[], commitId, policy): Graph` | `replayAt(logs: GraphLog[], commitId, policy): GraphStore` |
| 11 | Graph-only replay | available | **removed — `GraphStore` is the only replay target** |
| 12 | DEPENDENCY_ON_DEPRECATED | Core WARN violation | **removed** |
| 13 | SELF_LOOP | not detected | **ERROR** |

---

## Non-Breaking

- GraphStore, Graph (except `graphId` already required since v0.3), Commit schemas: **unchanged**
- All operations remain valid: `add_node`, `add_edge`, `supersede_edge`, `commit`
- Cross-graph edge support: **unchanged**
- Circular dependency detection: **unchanged** (extended to include supersedes cycles)
- A single-graph GraphStore remains fully valid — wrapping one Graph in a GraphStore
  is equivalent to v0.3 single-graph behavior in all respects.
  Note: the Graph-only replay API (`replay(ops[])`) has been removed;
  use `replay([{ graphId, ops }])` instead.
- The `supersedes` EdgeType is unchanged in meaning
- `DEPENDENCY_ON_SUPERSEDED` violation code is **retained**; only its detection logic changes
  (topology-derived, not stored Node attribute)

## Detailed Migration

### Change 1–2: Node.status Removed — Use effectiveStatus()

This is the core change of v0.4.

**v0.3 — stored state on node**

```json
{
  "type": "add_node",
  "node": {
    "id": "N:adr-001",
    "kind": "Decision",
    "status": "Active",
    "createdAt": "2026-01-01T00:00:00.000Z",
    "author": "alice"
  }
}
```

**v0.4 — Node carries no status**

```json
{
  "type": "add_node",
  "node": {
    "id": "N:adr-001",
    "kind": "Decision",
    "createdAt": "2026-01-01T00:00:00.000Z",
    "author": "alice"
  }
}
```

**v0.3 — checking supersession via stored attribute**

```typescript
if (node.status === "Superseded") { ... }
```

**v0.4 — checking supersession via topology**

```typescript
import { effectiveStatus } from "@decisiongraph/core";

if (effectiveStatus(store, nodeId) === "Superseded") { ... }
```

`effectiveStatus` is a pure function of the GraphStore.
It reads only edge topology — never any stored Node field.
It is cross-graph aware and replay-safe by construction.

**v0.3 — expressing supersession by mutating Node status (INVALID in v0.4)**

```json
{ "id": "N:adr-001", "status": "Superseded" }
```

**v0.4 — expressing supersession by adding a supersedes edge**

```json
{
  "type": "add_edge",
  "edge": {
    "id": "E:adr-003-sup-adr-001",
    "type": "supersedes",
    "from": "N:adr-003",
    "to": "N:adr-001",
    "status": "Active",
    "createdAt": "2026-02-01T00:00:00.000Z",
    "author": "alice"
  }
}
```

The existence of an Active `supersedes` edge targeting a Node is the **sole authority**
for its superseded status.

### Change 3: Deprecated Removed from Core

`Deprecated` is no longer a Core concept.

**v0.3**

```typescript
node.status === "Deprecated"  // Core-level lifecycle state
// DEPENDENCY_ON_DEPRECATED   // Core WARN violation
```

**v0.4**

`Deprecated` does not exist in the Core vocabulary.
Any deprecation semantics your domain requires belong in:

- extension policies (`@decisiongraph/policies` or your own)
- `payload` fields on Nodes or Edges
- domain-specific conventions above the kernel

If you rely on `DEPENDENCY_ON_DEPRECATED` violations, migrate them to an extension policy
that reads `payload` or a domain-specific edge type.

---

### Change 4: `overrides` Removed from EdgeType

**v0.3**

```typescript
type EdgeType = "depends_on" | "supports" | "refutes" | "overrides" | "supersedes";
```

**v0.4**

```typescript
type EdgeType = "depends_on" | "supports" | "refutes" | "supersedes";
```

Any edge with `type: "overrides"` will be rejected as `INVALID_EDGE_TYPE`.

**Migration options:**

- Replace with `supersedes` if the intent was structural supersession.
- Remove if the intent was deprecation signalling (now an extension concern).
- If the semantic distinction between "overrides" and "supersedes" matters in your domain,
  encode it in `payload` and handle it in an extension policy.

---

### Change 5: Edge Status — Deprecated Removed

**v0.3**

```typescript
type EdgeStatus = "Active" | "Superseded" | "Deprecated";
```

**v0.4**

```typescript
type EdgeStatus = "Active" | "Superseded";
```

Any edge with `status: "Deprecated"` will be rejected as `INVALID_EDGE_STATUS`.
Retire such edges via `supersede_edge` before migrating,
or remove them if they carried only deprecation intent.

---

### Change 6–7: Policy Interface — GraphStore-Based

**v0.3**

```typescript
interface Policy {
  validateOperation(graph: Graph, op: Operation): Violation[];
  validateGraph(graph: Graph): Violation[];
}
```

**v0.4**

```typescript
interface Policy {
  validateOperation(store: GraphStore, graphId: GraphId, op: Operation): Violation[];
  validateStore(store: GraphStore): Violation[];
}
```

Update all custom `Policy` implementations to accept `GraphStore` + `GraphId`.
The full store is required for cross-graph validation and `effectiveStatus` derivation.

---

### Change 8: ApplyResult — .graph → .store

**v0.3**

```typescript
const result = applyBatch(graph, ops, policy);
result.graph;  // Graph
```

**v0.4**

```typescript
const result = applyBatch(store, graphId, ops, policy);
result.store;  // GraphStore
```

---

### Change 9–10: replay / replayAt Signatures

**v0.3**

```typescript
replay(ops: Operation[], policy: Policy): Graph
replayAt(ops: Operation[], commitId: string, policy: Policy): Graph
```

**v0.4**

```typescript
type GraphLog = { graphId: GraphId; ops: Operation[] };

replay(logs: GraphLog[], policy: Policy): GraphStore
replayAt(logs: GraphLog[], commitId: string, policy: Policy): GraphStore
```

**Replay ordering contract (v0.4) — Normative:**

Logs MUST be provided in deterministic chronological order.
The array position of each log defines the replay time axis.
`createdAt` timestamps are preserved for audit but do not drive replay ordering.
Callers are responsible for sorting logs before passing them to `replay()` or `replayAt()`.
This contract is normative. The kernel does not sort logs.

```typescript
// v0.3
const state = replay(allOps, policy);

// v0.4
const logs: GraphLog[] = [
  { graphId: asGraphId("G:security"),  ops: securityOps },
  { graphId: asGraphId("G:adr-001"),   ops: adrOps },
];
const store = replay(logs, policy);
```

---

### Change 11: DEPENDENCY_ON_DEPRECATED Removed

The `DEPENDENCY_ON_DEPRECATED` violation code no longer exists in Core.

Remove any test assertions or CI rules that expect this code.
If your domain requires deprecation warnings, implement them in an extension policy.

---

### Change 12: SELF_LOOP Now Detected as ERROR

**v0.4** rejects any edge where `from === to` with `SELF_LOOP` ERROR.

```typescript
// Now rejected:
{ type: "add_edge", edge: { from: "N:a", to: "N:a", ... } }
// → SELF_LOOP ERROR
```

If your data contains self-loops, remove them before migrating.

## replayAt Boundary Semantics

`replayAt` uses a **store-wide boundary** (Constitution v0.4, Section 3.3).

Once the target `commitId` is encountered in any log,
all remaining ops across all logs are dropped:

```
logs = [
  G1: [add N:base, commit C:g1],
  G2: [add N:replacement, add supersedes-edge, commit C:g2]
]

replayAt(logs, "C:g1"):
  G1: add N:base ✓, commit C:g1 ✓ → stop
  G2: (not applied)
  → effectiveStatus(N:base) = "Active"

replayAt(logs, "C:g2"):
  G1: full ✓
  G2: full ✓
  → effectiveStatus(N:base) = "Superseded"
```

This guarantees that `effectiveStatus` is always consistent with
the state of the graph at the specified boundary — no additional mechanism required.

## Migration Checklist

### Log Files / Decision Data

- [ ] Remove `status` field from all Node records
- [ ] Replace any stored `status: "Superseded"` with a `supersedes` edge in a new Graph
- [ ] Remove or replace all edges with `type: "overrides"` (see Change 4)
- [ ] Remove or retire all edges with `status: "Deprecated"` (see Change 5)
- [ ] Verify no Node IDs, Edge IDs, or commitIds are duplicated across the GraphStore

### TypeScript Implementation

- [ ] Update all `Policy` implementations to `validateOperation(store, graphId, op)` / `validateStore(store)`
- [ ] Replace `result.graph` with `result.store` at all `apply` / `applyBatch` call sites
- [ ] Update `replay` / `replayAt` calls to pass `GraphLog[]` and receive `GraphStore`
- [ ] Replace `node.status === "Superseded"` with `effectiveStatus(store, nodeId) === "Superseded"`
- [ ] Remove all references to `NodeStatus` type (deleted from Core)
- [ ] Remove `"Deprecated"` from any `EdgeStatus` usage
- [ ] Remove `"overrides"` from any `EdgeType` usage

### Tests

- [ ] Remove fixtures that set `node.status` (schema now rejects this field)
- [ ] Remove test assertions expecting `DEPENDENCY_ON_DEPRECATED`
- [ ] Add test assertions using `effectiveStatus()` for supersession checks
- [ ] Verify `replayAt` tests pass `GraphLog[]` and assert on `GraphStore`
- [ ] Add self-loop edge to rejection test suite if not already present

### CI / Scripts

- [ ] Update `validate-decisions.mjs` or equivalent:
  - Pass `graphId` per log to `applyBatch(store, graphId, ops, policy)`
  - Accumulate into `GraphStore` across files
  - Use `lintStore(store, policy)` for final validation

## Automated Migration (Log Files)

Node `status` field removal can be automated:

```typescript
function migrateNodeToV04(
  node: Record<string, unknown>
): Record<string, unknown> {
  const { status, ...rest } = node;
  if (status !== undefined && status !== "Active") {
    // status was "Superseded" or "Deprecated" — caller must add a supersedes edge
    throw new Error(
      `Node '${(node as { id?: string }).id ?? "(unknown)"}' has status='${status}'. ` +
      `Add a supersedes edge targeting this Node, then re-run migration.`
    );
  }
  return rest; // strip "Active" status silently
}
```

Superseded Node state **cannot be automated** — it requires a new `supersedes` edge,
which requires knowing which Node supersedes it. Manual authorship is required.
This is intentional: the author of the supersession edge must be a real identity
(Constitution v0.4, Section 4.1).

## One-Line Summary of the Change

> v0.4 removes the last stored mutable state from Nodes.
> The kernel now knows exactly two things: a Node exists, and whether it has been superseded.
> Both are structural facts. Neither requires mutation.
