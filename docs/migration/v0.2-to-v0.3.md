# DecisionGraph Core v0.3 — Migration Guide

**From:** Constitution v0.2 (single-graph model) **To:** Constitution v0.3 (multi-graph / GraphStore model) **Purpose:** This document describes what changed, why it changed, and how to migrate.

---

## Background: Why v0.3?

During CLI validation work, a fundamental design constraint was identified. Cross-file `depends_on` references between committed Graphs were being rejected by the kernel as `IMMUTABLE_AFTER_COMMIT`:

```
✖ [02-adr-depends-on-security.decisionlog.json] op rejected: add_node
    IMMUTABLE_AFTER_COMMIT: graph is immutable after commit
✖ [02-adr-depends-on-security.decisionlog.json] op rejected: add_edge
    IMMUTABLE_AFTER_COMMIT: graph is immutable after commit
```

The root cause: v0.2 treats **one file = one graph = one accumulating state**. Once a Graph is committed, nothing can be added — including nodes from another file that legitimately depend on it.

This is not a bug. It is a design constraint that makes the core use case impossible:

> "Who decided what, under which premises, and how does that structurally relate to other decisions?"

Cross-graph references are not a workaround. They are the point. v0.3 resolves this at the kernel level by introducing `GraphStore`.

---

## Breaking Changes Summary

|#|Change|v0.2|v0.3|
|---|---|---|---|
|1|Top-level container|`Graph`|`GraphStore`|
|2|`graphId` on log files|absent|**REQUIRED**|
|3|`graphId` on `Graph` type|absent|**REQUIRED**|
|4|ID uniqueness scope|per-Graph|**GraphStore-wide**|
|5|Cross-graph edges|rejected|**supported & validated**|
|6|`applyBatch` signature|`(Graph, ops, policy)`|`(GraphStore, graphId, ops, policy)`|
|7|`lint` signature|`(Graph, policy)`|`(GraphStore, graphId, policy)`|
|8|`replay` signature|`(ops, policy)`|`(GraphLog[], policy)`|
|9|`lintStore`|absent|**NEW**|
|10|`resolveNode` / `resolveEdge`|absent|**NEW**|

---

## Non-Breaking (Unchanged)

- Node / Edge / Commit schemas are **unchanged**
- Status vocabulary (`Active | Superseded | Deprecated`) is **unchanged**
- Supersession model (`supersede_edge`) is **unchanged**
- All v0.2 constitutional guarantees (authorship, immutability, determinism) are **preserved**
- A single-graph `GraphStore` is **fully v0.2 compatible in behavior**

## Detailed Migration

### Change 1: GraphStore as Top-Level Container

**v0.2**

```typescript
// Graph is the top-level unit
const graph = emptyGraph();
const result = applyBatch(graph, ops, policy);
```

**v0.3**

```typescript
// GraphStore is the top-level unit
const store = emptyStore();
const result = applyBatch(store, "G:my-graph" as GraphId, ops, policy);
```

A `GraphStore` containing a single Graph behaves identically to v0.2. The only difference is the wrapping container and the required `graphId`.

---

### Change 2: `graphId` Required on Log Files

**v0.2 log file**

```json
{
  "version": "0.2",
  "ops": [ ... ]
}
```

**v0.3 log file**

```json
{
  "version": "0.3",
  "graphId": "G:security-policy",
  "ops": [ ... ]
}
```

Every log file **MUST** declare which Graph its operations belong to. The kernel uses `graphId` to route operations to the correct Graph within the GraphStore.

---

### Change 3: `graphId` Required on `Graph` Type

**v0.2**

```typescript
type Graph = {
  nodes: Record<string, Node>;
  edges: Record<string, Edge>;
  commits: Commit[];
};
```

**v0.3**

```typescript
type Graph = {
  graphId: GraphId;    // REQUIRED
  nodes: Record<string, Node>;
  edges: Record<string, Edge>;
  commits: Commit[];
};
```

---

### Change 4: ID Uniqueness is Now GraphStore-Wide

**v0.2:** Node IDs, Edge IDs, and commitIds must be unique within a single Graph.

**v0.3:** Node IDs, Edge IDs, and commitIds must be unique across the **entire GraphStore**.

```
// v0.2: this was allowed (same ID in different files)
Graph A: Node "N:001"
Graph B: Node "N:001"  ← OK in v0.2, ERROR in v0.3

// v0.3: all IDs must be globally unique
Graph A: Node "N:sec-001"
Graph B: Node "N:adr-001"  ← correct
```

**Migration action:** Audit all Node IDs, Edge IDs, and commitIds across all log files and ensure no duplicates exist across files.

---

### Change 5: Cross-Graph Edges Are Now Supported

**v0.2:** An Edge whose `to` node does not exist in the same Graph causes `EDGE_NOT_FOUND`.

**v0.3:** The kernel resolves `from` / `to` across the entire GraphStore. If the referenced Node exists in any Graph, the edge is valid. If it cannot be resolved anywhere, `EDGE_NOT_RESOLVED` is raised.

```json
// v0.3: this is now valid
{
  "version": "0.3",
  "graphId": "G:adr-001",
  "ops": [
    {
      "type": "add_edge",
      "edge": {
        "id": "E:adr-001-depends-sec",
        "type": "depends_on",
        "from": "N:adr-001",
        "to": "N:sec-policy-001",   // ← lives in G:security-policy
        "status": "Active",
        "createdAt": "2026-02-01T10:00:00.000Z",
        "author": "dev-team-alpha"
      }
    }
  ]
}
```

The immutability of `N:sec-policy-001` is governed by its owning Graph (`G:security-policy`).

---

### Change 6–8: Kernel Function Signatures

**applyBatch**

```typescript
// v0.2
applyBatch(graph: Graph, ops: Operation[], policy: Policy): ApplyResult

// v0.3
applyBatch(store: GraphStore, graphId: GraphId, ops: Operation[], policy: Policy): ApplyResult
// ApplyResult.graph → ApplyResult.store
```

**lint**

```typescript
// v0.2
lint(graph: Graph, policy: Policy): LintResult

// v0.3
lint(store: GraphStore, graphId: GraphId, policy: Policy): LintResult  // single graph
lintStore(store: GraphStore, policy: Policy): LintResult               // entire store (NEW)
```

**replay**

```typescript
// v0.2
replay(ops: Operation[], policy: Policy): Graph

// v0.3
type GraphLog = { graphId: GraphId; ops: Operation[] };
replay(logs: GraphLog[], policy: Policy): GraphStore
```

---

### Change 9–10: New Kernel Operations

**lintStore** — validates the entire GraphStore including cross-graph edges and circular dependencies.

```typescript
const result = lintStore(store, policy);
// checks: EDGE_NOT_RESOLVED, CIRCULAR_DEPENDENCY (across graphs)
```

**resolveNode / resolveEdge** — locate any Node or Edge across all Graphs.

```typescript
const resolved = resolveNode(store, "N:sec-policy-001" as NodeId);
// → { graphId: "G:security-policy", node: Node } | null
```

## Migration Checklist

### Log Files

- [ ] Update `"version"` from `"0.2"` to `"0.3"`
- [ ] Add `"graphId"` to every log file (unique, stable, non-empty)
- [ ] Audit all Node IDs, Edge IDs, commitIds for GraphStore-wide uniqueness

### Implementation

- [ ] Replace `emptyGraph()` with `emptyStore()`
- [ ] Add `graphId` parameter to `applyBatch`, `lint`, `replay` calls
- [ ] Replace `result.graph` with `result.store`
- [ ] Add `graphId` field to `Graph` type instantiations
- [ ] Update `Policy.validateOperation` signature to accept `GraphStore` + `graphId`
- [ ] Update `Policy.validateGraph` to `Policy.validateStore`
- [ ] Add `lintStore` calls where cross-graph validation is needed

### CI / Scripts

- [ ] Update `validate-decisions.mjs` or equivalent to pass `graphId` per log
- [ ] Ensure `decisiongraph lint ./decisions/` uses `lintStore` for final validation

## Automated Migration (Log Files)

```typescript
// Minimal automated migration for log files
function migrateLogToV03(
  log: { version: string; ops: unknown[] },
  graphId: string
): { version: "0.3"; graphId: string; ops: unknown[] } {
  if (log.version !== "0.2") {
    throw new Error(`Expected version 0.2, got ${log.version}`);
  }
  return {
    version: "0.3",
    graphId,
    ops: log.ops,
  };
}
```

Node IDs, Edge IDs, and commitId uniqueness across the GraphStore **cannot be automated** — manual audit is required.

## New Error Codes in v0.3

|Code|Description|
|---|---|
|`GRAPH_ID_REQUIRED`|`graphId` is missing or empty on a log or Graph|
|`GRAPH_ID_DUP`|`graphId` already exists in the GraphStore|
|`NODE_ID_DUP`|Node ID already exists in the GraphStore (store-wide, was per-graph in v0.2)|
|`EDGE_ID_DUP`|Edge ID already exists in the GraphStore (store-wide, was per-graph in v0.2)|
|`EDGE_NOT_RESOLVED`|Cross-graph edge endpoint cannot be resolved in any Graph|
|`CIRCULAR_DEPENDENCY`|Circular dependency detected within or across Graphs|
