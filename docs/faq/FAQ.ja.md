# FAQ — DecisionGraph Core（日本語）

Status: Non-normative. In case of conflict, Constitution v0.2 prevails.

> ⚠️ この文書は FAQ.md（英語正本）の参考訳・補足です。
> 仕様上の正式な判断は英語版および Constitution v0.2 を正とします。

---

## DecisionGraph Core はデータベースですか？

いいえ。

DecisionGraph Core は、
「どこにどう保存するか」を決めるシステムではありません。

それが定義するのは：

- 決定とは何か
- 決定がどう関係づけられるか
- それが有効か・再生可能か・監査可能か

という **構造と規則**です。

保存先は、ファイルでも DB でも Git でも構いません。
重要なのは「構造が壊れないこと」です。

---

## ナレッジグラフとは違うのですか？

はい、明確に違います。

一般的なナレッジグラフは：

- 事実
- 概念
- 推論結果

を扱います。

DecisionGraph Core が扱うのは：

- 人間が「こう決めた」と明示した判断
- その判断が、何に依存し、何を上書きしたか

です。

意味の補完や推論は一切行いません。

---

## ブロックチェーンの一種ですか？

いいえ。

DecisionGraph Core は：

- 分散合意
- マイニング
- トークン経済
- ネットワーク信頼

を提供しません。

不変性は **合意ではなく、規則で守られます**。

ブロックチェーンと組み合わせることは可能ですが、
必須ではありません。

---

## なぜ AI をカーネルに入れないのですか？

AI は本質的に確率的だからです。

DecisionGraph Core が保証したいのは：

- 同じ入力は同じ結果になる
- 過去を正確に再生できる
- 後から説明できる

という性質です。

AI をカーネル内部に組み込むと、これらの保証は成立しません。

AI は：
- 提案
- 下書き
- 補助

として **外側に置く**のが正しい位置づけです。

---

## なぜ author（作者）が必須なのですか？

作者のいない決定は、責任を持てないからです。

DecisionGraph Core は、
「判断をした主体が誰か」を保存します。

正しいかどうかは問いません。
ただ **誰が決めたか** を消させません。

---

## なぜコミット後に変更や削除ができないのですか？

判断の履歴を壊さないためです。

変更や削除を許すと：

- 監査ができない
- 過去が再生できない
- 責任の所在が曖昧になる

だから、変更は **supersession（上書き関係の追加）**
として表現します。

昔の判断は「無かったこと」にされません。
「当時はこうだった」と残ります。

---

## エンドユーザー向けのツールですか？

いいえ。

DecisionGraph Core は **インフラ**です。

ユーザー体験は、その上に作るものです。
このプロジェクト自体は、人に直接触らせる前提ではありません。

---

## 法務・医療・金融などにも使えますか？

使えます。

ただし、意味づけやルールは
ドメイン側が定義してください。

DecisionGraph Core は：
- 中立
- 最小
- 再利用可能

であることを優先しています。

---

## Git やドキュメントと何が違うのですか？

Git は「ファイルの履歴」を記録します。
DecisionGraph Core は「判断の構造」を記録します。

文章は理由を説明しますが、
DecisionGraph Core は **結果の構造**を保存します。

これにより：

- 判断の依存関係
- 上書き関係
- 有効性

を機械的に扱えるようになります。

---

## 人間の判断を置き換えるものですか？

いいえ。

DecisionGraph Core は判断しません。
記録するだけです。

判断を自動化するのではなく、
**判断を説明可能な形で残す**ためのものです。
